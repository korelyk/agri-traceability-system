## 第3页：研究背景

### 现实问题

- 农产品供应链环节多，信息分散
- 传统溯源系统依赖中心化数据库，可信性不足
- 消费者往往只能看到记录，无法判断记录是否可靠
- 传统二维码防伪容易被复制，缺少可信校验手段

### 选题目标

- 建立一个可运行的农产品溯源原型系统
- 将数字签名用于来源校验
- 将轻量级区块链用于防篡改校验
- 提供扫码查询与可视化展示能力

## 第4页：研究意义

### 理论意义

- 说明区块链与数字签名在溯源场景中的结合方式
- 验证“业务系统 + 链上校验层”的原型设计思路

### 实践意义

- 为消费者提供更直观的溯源查询手段
- 为食品安全监管提供可解释的技术方案
- 为工程开发型毕业设计提供完整原型案例

## 第5页：技术路线

- 前端 Web：Vue 3 + Element Plus
- 移动端：uni-app H5
- 后端：Spring Boot 2.7 + MyBatis-Plus
- 数据库：MySQL 8
- 区块链核心：Java 自实现轻量级区块链
- 安全：RSA 数字签名、BCrypt、JWT、私钥应用层加密
- 部署：Docker Compose + Nginx

答辩时强调：

- 当前是单机原型
- 不是多节点联盟链
- 重点是“可运行、可验证、可演示”

## 第6页：为什么引入区块链和数字签名

### 区块链的作用

- 通过前后区块哈希关联提高篡改成本
- 通过 PoW 模拟区块确认过程
- 为公开查询提供链上校验依据

### 数字签名的作用

- 证明记录来源于合法主体
- 防止记录在传输或写入过程中被伪造
- 提供“不可抵赖”的解释能力

## 第8页：系统总体架构

### 分层说明

1. 展示层
Web 前端负责管理功能和统计展示。

2. 移动展示层
uni-app H5 负责手机扫码查询和简化业务入口。

3. 业务服务层
Spring Boot 提供用户、产品、溯源、统计和公开查询接口。

4. 数据与校验层
MySQL 保存业务数据，轻量级区块链保存校验相关交易与区块信息。

## 第9页：核心业务流程

### 产品注册

- 生产者登录
- 填写产品基础信息
- 生成产品记录和首条链上交易
- 返回产品 ID 和二维码

### 溯源记录新增

- 操作者提交记录
- 后端生成交易摘要
- 使用私钥签名
- 验签通过后写入链和数据库

### 公开查询

- 用户扫码进入查询页
- 查询数据库记录
- 对比链上交易
- 返回验证通过或异常提示

## 第10页：区块链设计

### 区块字段

- 当前区块哈希
- 前一区块哈希
- Merkle Root
- 时间戳
- Nonce
- 难度值
- 交易列表

### PoW 说明

- 通过不断尝试 `nonce` 计算哈希
- 满足前导零难度要求后才视为区块有效
- 用于模拟区块确认过程，而非真实挖矿经济系统

---

## 第11页：安全设计

### 已实现

- 密码 BCrypt 哈希存储
- 登录后返回 JWT
- 私钥应用层加密存储
- 溯源交易 RSA 签名与验签
- 公开查询时进行一致性校验
- 链状态文件持久化恢复

### 当前边界

- 单机原型，不是多节点联盟链
- 没有独立审计告警中心
- 权限模型以答辩演示可用为主
