# 农产品防伪溯源系统 - 项目完整思路

## 1. 项目定位

本项目是一个面向本科毕业设计答辩的工程开发型原型系统，主题为“基于区块链与数字签名的农产品防伪溯源系统设计与实现”。

它不是多节点联盟链平台，也不是面向生产环境的大规模商用系统，而是一个围绕以下目标构建的可运行原型：

- 演示农产品从生产到加工环节的基础溯源流程
- 演示 RSA 数字签名在数据来源校验中的作用
- 演示轻量级区块链在防篡改校验中的应用方式
- 提供 Web 端、移动端 H5 和二维码公开查询能力
- 支撑毕业论文、答辩展示和现场演示

## 2. 问题背景

传统农产品溯源系统通常采用中心化数据库，存在几类典型问题：

- 数据容易被单点管理员修改，可信性不足
- 溯源记录分散在多个环节，信息透明度不高
- 消费者只能看到“有记录”，难以判断“记录是否可信”
- 传统二维码防伪容易被复制，缺少可信校验手段

因此，本项目选择“业务数据 + 链上校验”的组合方案，不追求完整联盟链治理，而是聚焦在“毕业设计场景下可验证、可演示、可解释”的技术实现。

## 3. 总体方案

### 3.1 技术路线

- 前端 Web：Vue 3、Vuex、Vue Router、Element Plus
- 移动端：uni-app H5
- 后端：Spring Boot 2.7、MyBatis-Plus
- 数据库：MySQL 8
- 区块链核心：Java 自实现轻量级区块链
- 密码学能力：SHA-256、RSA 数字签名、BCrypt、应用层私钥加密
- 部署方式：Docker Compose + Nginx

### 3.2 存储策略

系统采用“双轨存储”：

- MySQL 保存用户、产品、溯源记录等业务数据，便于查询与展示
- 轻量级区块链保存交易摘要、签名、哈希链关系，用于数据一致性校验

这意味着系统更像“业务系统 + 链上校验层”，而不是把全部业务逻辑都塞进链上。

### 3.3 当前可信边界

本项目当前可信模型成立在以下前提下：

- 区块链状态文件未被越权删除或替换
- 应用服务与数据库运行在受控部署环境中
- 关键交易经过 RSA 签名校验后再写入链上
- 链上交易与数据库记录通过交易 ID 进行一致性比对

因此，答辩时最稳妥的表述是：

“本系统通过数字签名和轻量级区块链提升溯源记录的可验证性与防篡改解释能力，但当前仍属于单机原型，不等同于多节点联盟链。”

## 4. 系统架构

### 4.1 分层结构

系统采用前后端分离的 B/S 架构，并保留一个移动端 H5 入口：

1. 展示层
Web 前端负责后台管理、产品注册、溯源录入、统计分析和区块浏览。

2. 移动展示层
uni-app H5 负责手机扫码查询和基础业务演示。

3. 业务服务层
Spring Boot 提供用户注册登录、产品管理、溯源记录、统计分析、公开查询等接口。

4. 数据与校验层
MySQL 负责结构化业务数据持久化，Java 轻量级区块链负责交易打包、PoW 挖矿、链状态恢复和一致性校验。

### 4.2 核心模块

- 用户模块：注册、登录、JWT 鉴权、角色区分
- 产品模块：注册产品、展示产品列表、查看产品详情、生成二维码
- 溯源模块：新增溯源记录、公开查询、链上校验展示
- 区块链模块：交易池、区块打包、哈希链校验、区块统计
- 安全模块：密码哈希、私钥加密、数字签名与验签
- 部署模块：Docker Compose 部署 MySQL、后端和前端

## 5. 核心业务流程

### 5.1 用户注册与登录

- 用户注册时写入基本资料和角色信息
- 后端为用户生成 RSA 公私钥对
- 公钥直接保存，私钥经过应用层加密后保存
- 用户密码通过 BCrypt 哈希后入库
- 登录时校验哈希密码，登录成功后返回 JWT

### 5.2 产品注册流程

- 生产者登录后填写产品名称、类别、产地和描述
- 系统创建产品记录，并生成首条“生产注册”类型交易
- 交易携带操作者身份、公钥和数字签名
- 交易进入待处理池，达到条件后打包进区块
- 产品生成对应二维码，供后续公开查询使用

### 5.3 溯源记录新增流程

- 已登录用户选择产品并填写操作类型、地点和详情
- 后端根据当前用户身份生成交易内容
- 后端使用解密后的私钥对交易摘要签名
- 链上验签通过后，交易写入区块链状态
- 同步写入 MySQL 业务表，供前端查询展示

### 5.4 公开查询流程

- 消费者通过产品 ID 或二维码访问公开查询页
- 系统读取产品信息和溯源记录
- 同时查询链上对应交易
- 对数据库记录与链上交易进行一致性比对
- 前端展示“数据验证通过”或“数据存在异常”

当前系统实现的是“异常校验提示”，不是独立的告警中心。

## 6. 区块链设计思路

### 6.1 为什么引入轻量级区块链

项目引入区块链的目标不是发币，也不是模拟完整公链生态，而是解决两个问题：

- 为溯源记录提供链式防篡改结构
- 为答辩展示提供“为什么这不是普通数据库”的技术解释

### 6.2 区块结构

每个区块主要包含：

- 区块索引
- 前一区块哈希
- 当前区块哈希
- Merkle Root
- 时间戳
- 难度值
- Nonce
- 交易列表

### 6.3 工作量证明

系统采用简化的 PoW 机制：

- 区块打包后不断尝试不同的 `nonce`
- 重复计算区块哈希
- 当哈希满足前导零难度要求时，视为“挖矿成功”

它的作用不是追求真实矿工博弈，而是：

- 增加伪造新区块的计算成本
- 让区块生成具备可解释的“确认过程”
- 强化“区块不是随便写进去的”这一答辩表达

### 6.4 链状态持久化

项目最终版本已支持链状态文件持久化：

- 启动时读取状态文件恢复链
- 新区块写入后同步更新状态文件
- 服务重启后不会丢失区块链状态

这一步解决了早期原型中“服务重启只剩创世块”的问题。

## 7. 安全设计

### 7.1 已实现的安全措施

- 用户密码采用 BCrypt 哈希存储
- 登录后采用 JWT 进行接口鉴权
- 用户私钥采用应用层加密保存
- 溯源交易采用 RSA 数字签名
- 上链前进行签名验证
- 公开查询时进行链上与库内数据一致性校验

### 7.2 当前仍然保留的边界

- 当前是单机原型，没有多节点共识网络
- 链状态文件仍在应用侧托管，不是去中心化分布式存储
- 没有独立告警系统、审计中心和操作审批流
- 权限控制以答辩演示可用为主，不是企业级 RBAC 完整实现

## 8. 模块价值说明

### 8.1 对消费者

- 可以扫码查看产品来源和流转信息
- 可以直观看到“验证通过/异常”提示

### 8.2 对生产与加工主体

- 可以记录关键操作节点
- 可以用统一页面维护产品与溯源信息

### 8.3 对答辩场景

- 能完整展示“注册产品 -> 添加记录 -> 查询验证”的闭环
- 能解释数字签名、PoW、哈希链、链上校验等核心概念

## 9. 项目亮点

- 不是只做页面，而是完成了完整前后端原型和真实部署
- 将 RSA 数字签名与链上交易校验结合起来
- 增加了私钥加密存储和链状态持久化
- 支持二维码公开查询与移动端访问
- 文档、代码、部署环境已经基本统一

## 10. 局限与后续改进

适合在答辩中主动说明的不足如下：

- 当前为单机原型，不是多节点联盟链
- 统计分析偏演示型，缺少更细颗粒度报表
- 没有接入 IoT 自动采集设备
- 没有实现完善的企业级审计告警链路
- 自动化测试覆盖面仍有继续扩展空间

后续可扩展方向：

- 升级为联盟链或多节点验证模式
- 引入独立审计日志和异常告警模块
- 补充更细化的角色权限模型
- 接入传感器与自动采集设备
- 增加更完整的接口测试与端到端测试

## 11. 适合答辩时的总结口径

本项目完成了一个可运行、可部署、可演示的农产品防伪溯源原型系统。系统以 MySQL 承载业务数据，以轻量级区块链承载校验逻辑，通过 RSA 数字签名和哈希链结构提高了溯源数据的可验证性，并通过二维码公开查询增强了用户可感知的使用效果。虽然当前版本仍属于单机原型，但已经具备完整的工程闭环，能够满足本科毕业设计关于系统原型、需求分析、设计实现和测试验证的基本要求。
